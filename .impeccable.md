## Design Context

### Users
Developers actively preparing for technical interviews — practicing algorithms, debugging realistic scenarios, and building confidence through hands-on challenges in live container environments. They arrive already motivated but anxious; the UI's job is to meet them with calm competence. They're comfortable with technical tooling (terminal, IDE, code editors) and will immediately notice if something feels off or condescending.

### Brand Personality
**Three words**: Approachable, technical, encouraging.

The name says it all: *real work*. This isn't a toy or a quiz — it's a place where developers do the actual thing. The interface should feel like a well-built tool made by engineers who care, not a gamified learning app or a cold judge of merit. It has warmth without being cutesy, precision without being sterile.

### Aesthetic Direction
**Dark-first**, light-mode supported. The primary experience lives in dark mode — deep backgrounds that echo the terminal and IDE environments users already live in, but warmer and more inviting. The teal-green primary palette (OKLCH-based, already defined in globals.css) bridges the technical and the human.

**References**: Vercel (precision, restraint, trust), LeetCode (familiar challenge format), HackerRank (structured rigor), Figma (professional tool feel, collaborative). Draw from the *feeling* of each — the purposefulness of Vercel, the task-focus of LeetCode — without replicating any single one.

**Anti-references**: Avoid: gamification chrome (badges everywhere, XP bars), neon-on-black "hacker" aesthetics, clunky LeetCode-style information density without breathing room, Figma's visual complexity transplanted wholesale.

**Visual tone**: Refined terminal. Think: the IDE you'd build if you had taste. Generous whitespace (for a technical tool), purposeful motion, type that communicates hierarchy without shouting.

### Design Principles

1. **The work is the hero** — every layout decision should direct attention to the challenge content, the code, the problem. The UI is the stage, not the performer. Avoid decorative elements that compete for focus.

2. **Earn trust through precision** — developers are skeptical. Pixel-perfect alignment, consistent spacing, typographic care, and predictable interactions signal that this tool was built by people who take craft seriously.

3. **Encouraging without patronizing** — the interface should feel like a patient, confident mentor. Clear affordances, informative empty states, helpful error messaging. Never cold, never saccharine.

4. **Dark-first warmth** — embrace dark mode as the primary canvas, but warm it. The deep blue-green backgrounds (`oklch(0.14 0.02 200)`) with the phosphor-green primary (`oklch(0.72 0.17 155)`) should feel like a well-lit workspace, not a cave.

5. **Distinctly real_work** — inspired by the precision of Vercel, the familiarity of LeetCode, the tool-ness of Figma — but unmistakably its own. Make choices that would make a designer ask "what's that built with?"

### Technical Notes
- Tailwind CSS v4 with OKLCH design tokens in `app/globals.css`
- shadcn/ui component library (radix primitives + CVA variants)
- Inter (sans) + Geist Mono (code) — consider whether Inter is distinctive enough given the personality; could be revisited
- Responsive: mobile-first, Tailwind default breakpoints (sm/md/lg/xl)
- next-themes for dark/light switching
- No specific WCAG level required; maintain good baseline contrast and focus states
